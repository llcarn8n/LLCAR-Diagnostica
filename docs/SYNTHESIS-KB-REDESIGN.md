## 1. Противоречия между предложениями

### 1.1. Оффлайн: Амбиции vs Реальность

**Документ A** обещает Service Worker + IndexedDB с автокэшированием экстренных инструкций, чеклистов, последних 20 статей. Оценка сложности: «Высокая» (3-5 дней, Фаза 4).

**Документ B** предлагает полноценный офлайн-пакет на 29 МБ с предвычисленными векторами, ситуационными шаблонами, lite-БД. Оценка: ~20-24 часа разработки.

**Противоречие:** Документ A рассматривает оффлайн как «кэширование статей», а Документ B — как «полноценный офлайн-поисковый движок». Это два принципиально разных уровня.

**Вердикт:** Документ B переусложняет. Для Li Auto L7/L9 диагностики (нишевое приложение, десятки-сотни пользователей) офлайн-поиск с предвычисленными векторами — это overengineering. Документ A правильнее: простое кэширование ключевых статей через Service Worker достаточно. **НО** — ситуационные индексы из Документа B (`situation-index.json`) полезны и без офлайна: они решают задачу контекстных рекомендаций на сервере.

### 1.2. Навигация: 4 точки входа vs 5 табов

**Документ A** заменяет 5 табов (`all/systems/parts/manuals/dtc`) на 4 ситуационных точки входа: Срочное, Обслуживание, Моя машина, Изучить.

**Текущий код (C)** имеет `CATEGORIES = ['all', 'systems', 'parts', 'manuals', 'dtc']` с горизонтальными табами. Каждый таб запускает свой рендер (`renderLayerView`, `renderPartsView`, `renderDtcView`, etc).

**Противоречие:** A предлагает полную перестройку навигации. Это ломает весь `renderResults()` (200+ строк), `renderCategoryTabs()`, и все связанные рендер-функции.

**Вердикт:** Идея Документа A правильная — ситуационная навигация ближе к пользователю. НО полная переделка за 1-2 дня нереалистична для соло-разработчика. Компромисс: **оставить табы, но переименовать и реорганизовать** — `Urgent | Service | Systems | Parts | Browse`. Это на 80% даёт эффект без переписывания всей архитектуры рендеринга.

### 1.3. Контекстные рекомендации: Фронтенд vs Бэкенд

**Документ A** описывает блок «Топ-3 для вашей ситуации» на фронтенде, собирающий контекст из localStorage + event bus.

**Документ B** предлагает `SituationMatcher` — JS-класс с индексным матчингом и dot-product ранжированием, работающий полностью на клиенте.

**Противоречие:** A хочет серверный поиск с фильтрами (через `/search`), B хочет клиентский матчинг без сервера.

**Вердикт:** Для онлайн-режима (основной сценарий) — серверный подход из A проще и надёжнее. Клиентский `SituationMatcher` из B имеет смысл ТОЛЬКО для офлайна. Рекомендация: **сначала серверные рекомендации (A), потом при необходимости клиентский фоллбек (B)**.

### 1.4. Модель эмбеддингов

**Документ B** предлагает `multilingual-e5-small` (384d) для предвычисления офлайн-векторов, в дополнение к уже существующим pplx-embed (2560d) и BGE-M3 (ColBERT).

**Текущий код (C)** уже использует pplx-embed + BGE-M3 через серверный API.

**Противоречие:** Три модели эмбеддингов — это избыточно. multilingual-e5-small нужна только для офлайн-пакета.

**Вердикт:** НЕ добавлять третью модель. Для ситуационного индекса (situation-index.json) ML-модель вообще не нужна — достаточно rule-based тегирования из `auto_tag_article()`. Векторный ранжинг в офлайне — преждевременная оптимизация.

### 1.5. Чеклисты и шаблоны

**Документ A** описывает чеклисты подготовки к поездке, экстренные инструкции с пошаговыми действиями, поток «Что-то не так» (DTC -> Действие).

**Текущий код (C)** не имеет ничего подобного — только плоский список результатов поиска.

**Документ B** не затрагивает чеклисты.

**Вердикт:** Чеклисты — это КОНТЕНТ, а не код. Чтобы их реализовать, нужно: (1) создать шаблоны чеклистов в JSON, (2) написать рендер. Это ценно, но требует ручного написания контента. Для MVP можно захардкодить 3-5 ключевых чеклистов.

## 3. MVP vs Full

### MVP (2-3 дня) — 80% ценности

**Цель:** Существенно улучшить восприятие результатов поиска без переделки архитектуры.

```
День 1 (фронтенд, ~6 часов):
├── [1] Индикатор доверия: SOURCE_BADGES + trust_level (1-5)
│   → Добавить trust_level в существующий SOURCE_BADGES объект
│   → Рендер 5 точек в карточке результата (renderResultCard)
│   → CSS: .trust-dot { width:6px; height:6px; border-radius:50% }
│
├── [2] Индикатор перевода
│   → Условие: source_language !== user_language → жёлтая метка
│   → Данные уже есть в API response (source_language поле)
│
└── [3] Экстренный режим карточки
    → CSS-класс .kb-card--emergency { border: 2px solid #EF5350; background: rgba(239,83,80,0.08) }
    → Условие: has_warnings === true || severity >= 7
    → Приоритет в сортировке: экстренные всегда первые

День 2 (бэкенд скрипт + фронтенд, ~6 часов):
├── [4] build_situation_tags.py (Документ B, секция 5.2)
│   → Чистый SQL + regex, без ML. Занимает 30 сек.
│   → Создаёт situation_tags.json: situation_type, season, urgency для каждого chunk
│   → Добавить urgency + situation_type в ответ API /search
│
└── [5] Режимы карточек (список / карточки)
    → Два шаблона: compact (1 строка) и card (текущий, расширенный)
    → Переключатель в заголовке (иконки ≡ / ⊞)
    → localStorage для запоминания выбора

День 3 (интеграция + полировка, ~4 часа):
├── [6] Блок «Сейчас важно» — упрощённая версия
│   → Если есть activeVehicle в localStorage + текущий месяц
│   → Захардкоженные 2-3 рекомендации по сезону (зима: антифриз, аккумулятор; лето: кондиционер)
│   → Простой API-запрос /search с фильтром layer + season keyword
│
└── [7] Тесты и баги
```

**Что НЕ входит в MVP:**
- Полная смена навигации (4 точки входа)
- Контекстный движок рекомендаций
- Чеклисты
- Оффлайн
- SituationMatcher
- Предвычисленные векторы
- Блок «Разные взгляды»

### Full (2-3 недели)

```
Неделя 1: Навигация + Контекст
├── Новые 4 точки входа (переделка renderContent + renderResults)
├── Контекстный движок рекомендаций (серверный)
├── Ситуационный индекс (build_situation_index.py + API endpoint /recommendations)
├── Блок «Сейчас важно» с реальными DTC + пробегом
└── Блок «Разные взгляды»

Неделя 2: Чеклисты + 3D интеграция
├── JSON-шаблоны чеклистов (ТО-1..ТО-5, зимняя подготовка, дальняя поездка)
├── Рендер чеклистов с checkbox UI + localStorage для прогресса
├── Расширенный bottom sheet в 3D (статьи + запчасти + DTC)
├── Поток «DTC -> Действие» (экстренная карточка -> полная статья)
└── Подробный режим карточки (третий вид)

Неделя 3: Оффлайн
├── Service Worker для статических ресурсов
├── IndexedDB для экстренных инструкций + последних статей
├── build_offline_db.py (lite-БД, ~15 МБ)
├── Визуальная индикация оффлайн-доступности
└── Кнопка «Сохранить для оффлайн»
```

## 4. Техническая реальность

### 4.1. Что можно менять инкрементально (без переписывания)

| Изменение | Что затрагивает | Риск |
|-----------|----------------|------|
| Добавить `trust_level` в `SOURCE_BADGES` | Один объект, 20 строк | Нулевой |
| Добавить CSS-класс `.kb-card--emergency` | Только рендер карточки | Нулевой |
| Добавить индикатор перевода | Рендер карточки, ~10 строк | Нулевой |
| Добавить переключатель режимов (список/карточки) | Новая функция `renderCompactCard()` + переключатель | Низкий |
| Добавить `situation_type` и `urgency` в API response | Изменения в `server.py` — 1 SQL JOIN | Низкий |
| Добавить статичный блок «Сейчас важно» | Вставка HTML перед `#kb-results` | Низкий |

### 4.2. Что требует значительной переработки

| Изменение | Что ломается | Объём |
|-----------|-------------|-------|
| 4 точки входа вместо табов | `CATEGORIES`, `renderCategoryTabs()`, `renderResults()`, `activeCategory` state, все switch-кейсы по категориям | ~400 строк |
| Контекстный движок рекомендаций | Новый модуль, интеграция с localStorage, event bus, API | Новый файл ~200 строк |
| Чеклисты | Новый формат данных, рендер, localStorage для прогресса | Новый файл ~300 строк + JSON-контент |
| Оффлайн (Service Worker) | Новый SW файл, manifest, IndexedDB wrapper | ~500 строк нового кода |
| SituationMatcher (Документ B) | Новый модуль, бинарные данные, WASM (sql.js) | ~600 строк + pipeline скриптов |

### 4.3. Структура текущего кода (knowledge.js)

```
knowledge.js (1261 строк):
├── Константы: LAYER_COLORS, CATEGORIES, SOURCE_BADGES, SYSTEM_NAME_RU (~100 строк)
├── render() — точка входа, инициализация KB (~50 строк)
├── renderContent() — основной layout + HTML (~80 строк)
├── renderCategoryTabs() — горизонтальные табы (~30 строк)
├── renderResults() — диспетчер по activeCategory (~250 строк)
│   ├── default (search results) — список карточек
│   ├── 'systems' → renderLayerView()
│   ├── 'parts' → renderPartsView()
│   ├── 'manuals' → renderManualsList()
│   └── 'dtc' → renderDtcView()
├── Отдельные рендеры:
│   ├── renderLayerView() — системы по слоям (~100 строк)
│   ├── renderPartsView() — каталог запчастей (~150 строк)
│   ├── renderPartDetail() — детальная запчасть (~100 строк)
│   ├── renderManualsList() — список руководств (~80 строк)
│   ├── renderDtcView() — DTC коды (~100 строк)
│   └── renderArticle() — полная статья (~100 строк)
├── bindEvents() — обработчики событий (~80 строк)
└── Утилиты: _onComponentSelect, _onLangChange, updateTagFilter (~60 строк)
```

**Ключевой вывод:** `renderResults()` — это god-function на 250 строк с switch по категориям. Любая серьёзная переделка навигации требует рефакторинга этой функции. НО для MVP (индикаторы доверия, перевода, экстренный режим) достаточно изменить ТОЛЬКО рендер отдельной карточки — это 30-40 строк в каждом рендере.

### 4.4. knowledge-base.js — API-клиент

Клиент (493 строки) хорошо структурирован и НЕ ТРЕБУЕТ изменений для MVP. Все нужные методы уже есть:

- `search()` — возвращает `source_language`, `has_procedures`, `has_warnings`
- `getDTC()` — возвращает severity
- `searchParts()` — каталог запчастей
- `getChunk()` — полный чанк с переводами
- `_Cache` — LRU кэш работает

Единственное, что может понадобиться: метод `getRecommendations(context)` для контекстного движка (Tier 2).

## 5. Единая рекомендация

### Стратегия: «Улучшить карточки, не ломая архитектуру»

**Фаза 1 (MVP, 2-3 дня): Доверие + Экстренность**

Минимальные изменения в существующем коде, максимальный визуальный эффект:

1. **Расширить `SOURCE_BADGES`** — добавить поле `trustLevel: 1-5` к каждому источнику. Рендерить 5 точек в каждой карточке. Это 30 минут работы.

2. **Добавить индикатор перевода** — если `source_language !== 'ru'`, показать метку `[ZH->RU Автоматический перевод]`. Данные уже есть в ответе API. 30 минут.

3. **Добавить экстренный CSS-класс** — `.kb-card--emergency` с красной рамкой и крупным шрифтом. Условие: `has_warnings === true`. Сортировать экстренные наверх. 1 час.

4. **Запустить `build_situation_tags.py`** — автоматическая разметка всех 11,398 статей (situation_type, urgency, season). НЕ для офлайна, а для обогащения серверного ответа. Добавить `urgency` и `situation_type` в response `/search`. 3-4 часа.

5. **Добавить компактный режим списка** — второй шаблон рендера (1 строка: заголовок + мета + скор). Переключатель в заголовке. 2-3 часа.

6. **Статичный блок «Сейчас»** — над результатами, 2-3 карточки по сезону. Без сложной логики — просто API-запрос с сезонным ключевым словом. 2 часа.

**Фаза 2 (если будет время, +1 неделя): Навигация + Контекст**

Только ПОСЛЕ успешного MVP:

7. Рефакторинг `renderResults()` — разбить god-function на отдельные модули.
8. Переименовать табы: `Urgent | Service | Systems | Parts | Browse`.
9. Контекстный движок рекомендаций (серверный `/recommendations` endpoint).
10. Расширенный bottom sheet в 3D.

**Фаза 3 (отложить): Оффлайн**

НЕ делать сейчас. Причины:
- Аудитория мала (десятки пользователей)
- Service Worker + IndexedDB — серьёзная инфраструктура для поддержки
- PSV-пакет из Документа B (29 МБ, multilingual-e5-small) — overengineering для текущего масштаба
- Офлайн имеет смысл только если приложение станет реальным PWA с сотнями пользователей

**Что взять из каждого документа:**

| Документ | Что берём | Что НЕ берём |
|----------|-----------|-------------|
| A (UX) | Индикаторы доверия, перевода, экстренный режим, режимы карточек, блок «Сейчас важно» | Полную навигацию 4 точек входа (слишком много переделок), чеклисты (нужен контент), оффлайн-UI |
| B (Архитектура) | `auto_tag_article()` для разметки, ситуационный индекс (серверная версия) | PSV-пакет, SituationMatcher, multilingual-e5-small, бинарные векторы, articles-lite.db |
| C (Текущий код) | Всю архитектуру рендеринга, API-клиент, LRU-кэш, event bus | Плоскую сортировку без приоритетов, отсутствие визуального различения источников |

### Итоговая формула

> **Не переписывай. Обогащай.**
>
> Текущий код работает. API работает. Поиск работает.
> Задача MVP — сделать результаты поиска УМНЕЕ и ЧИТАБЕЛЬНЕЕ,
> не меняя то, как они получаются.

```javascript
function renderTrustDots(trustLevel) {
  const dots = [];
  for (let i = 1; i <= 5; i++) {
    const filled = i <= trustLevel;
    dots.push(`<span style="display:inline-block;width:6px;height:6px;border-radius:50;margin-right:2px;background:${filled ? 'var(--accent-primary)' : 'var(--border-default)'};"></span>`);
  }
  return dots.join('');
}
```

```javascript
function renderTranslationBadge(result, userLang) {
  if (!result.source_language || result.source_language === userLang) return '';
  const langMap = { zh: 'ZH', en: 'EN', ru: 'RU' };
  const from = langMap[result.source_language] || result.source_language.toUpperCase();
  const to = langMap[userLang] || userLang.toUpperCase();
  return `<span style="font-size:10px;padding:2px 6px;border-radius:4px;background:#FFF8E1;color:#F57F17;">${from} -> ${to}</span>`;
}
```
